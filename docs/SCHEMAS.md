# Table schemas

This document records the public-facing schemas used by the thesis workflows. The legacy scripts are preserved for provenance, so some historical files may contain extra columns. The columns below are the minimum stable fields that should be validated before running canonical steps.

## Chapter 4 sample metadata

Used by `legacy/chapter4/00_manifest_builder.py` and downstream PCR/non-PCR analyses.

| Column | Required | Meaning | Example |
|---|---:|---|---|
| `file_id` | yes | Unique sample/file identifier used to match FASTQ files | `H12` |
| `sample_name` | yes | Descriptive sample name | `MP10_12` |
| `species` | yes | Species or mixture label | `10_species_mix` |
| `n_individuals` | yes | Number of individuals/species represented | `10` |
| `locus` | yes | Amplicon locus, usually `12S` or `16S` | `12S` |
| `pcr` | yes | PCR status, normalized conceptually to yes/no | `yes` |

Canonical header:

```text
file_id\tsample_name\tspecies\tn_individuals\tlocus\tpcr
```

Notes:

- `file_id` must be unique.
- `locus` should be `12S` or `16S` in the main thesis analysis.
- `pcr` should be a clear yes/no value. Use lower-case `yes` and `no` for new metadata.
- Do not commit private sample sheets if they contain unpublished or sensitive metadata.

## Chapter 4 manifest

Generated by `legacy/chapter4/00_manifest_builder.py`.

| Column | Meaning |
|---|---|
| `file_id`, `sample_name`, `species`, `n_individuals`, `locus`, `pcr` | Metadata fields copied from the input sample table |
| `input_type` | Current public workflow assumes `SE_FASTQ` |
| `fastq_path` | Resolved local FASTQ path |
| `source_dir` | `raw` or `trimmed` |
| `match_rule` | Rule used to match metadata to FASTQ filename |
| `n_matches` | Number of matching candidate files |
| `all_matches` | Candidate paths retained for manual inspection |
| `status` | `OK`, `MISSING`, or `AMBIGUOUS` |

Rows with `MISSING` must not be used downstream. Rows with `AMBIGUOUS` should be manually checked before preprocessing.

## Chapter 4 sequence preprocessing outputs

Generated by `legacy/chapter4/01_preprocess_sequences_se.py`.

### Per-sample stats: `<file_id>_stats.tsv`

| Column | Meaning |
|---|---|
| `file_id`, `sample_name`, `species`, `n_individuals`, `locus`, `pcr` | Metadata copied from manifest |
| `Seq_ID` | MD5-based sequence ID |
| `Sequence` | Observed sequence |
| `Length` | Sequence length |
| `Count` | Read count for the sequence |
| `Rel_Abund` | Within-sample relative abundance |
| `Tag` | Top-rank or count-based filtering tags |

### `samples_summary.tsv`

Must contain at least:

```text
file_id, sample_name, species, n_individuals, locus, pcr, status, fastq_path, total_reads, unique_sequences
```

## Chapter 2–3 sample metadata

Used by `legacy/chapter2_3/03_build_master_long_abundance.py` and downstream Chapter 2–3 scripts.

| Column | Required | Meaning |
|---|---:|---|
| `sample_id` | yes | Sample identifier used to locate filtered FASTQ files |
| `marker` | yes | Marker/locus label, usually `12S` or `16S` |
| `group_name` | yes | Experimental group or mixture name |
| `sample_type` | yes | Design class, for example inter- or intra-mixture |
| `species_scope` | yes | Scope of species analysis |
| `is_core_analysis` | yes | Whether the sample belongs to the core analysis set |

Canonical header:

```text
sample_id\tmarker\tgroup_name\tsample_type\tspecies_scope\tis_core_analysis
```

## Schema alignment between workflows

Chapter 2–3 and Chapter 4 should not be forced into a single metadata table because they were designed for different analysis layers. Use this conceptual mapping only when writing documentation or harmonizing outputs:

| Chapter 2–3 | Chapter 4 | Comment |
|---|---|---|
| `sample_id` | `file_id` | Both identify input files/samples |
| `marker` | `locus` | 12S/16S marker field |
| `group_name` | `sample_name` or `species` | Depends on analysis context |
| `sample_type` | derived from `pcr` and design | Not a direct one-to-one mapping |
| `species_scope` | `species` | Similar but not identical semantics |
| `is_core_analysis` | no direct equivalent | Chapter 4 uses script/status separation instead |
